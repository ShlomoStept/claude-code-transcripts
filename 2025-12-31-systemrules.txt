## 1. Project Overview and Documentation Set

### What the Project Is

**claude-code-transcripts** is a Python CLI tool that converts Claude Code 
session files (JSON or JSONL format) into clean, mobile-friendly, paginated 
HTML pages. The tool serves as both a local transcript viewer and a 
publishing mechanism for sharing Claude Code interactions.

**Key Capabilities:**
- Convert local Claude Code sessions (stored as JSONL in 
`~/.claude/projects`)
- Import web sessions via Claude API
- Generate searchable, paginated HTML transcripts
- Publish transcripts to GitHub Gists with preview URLs
- Archive all local sessions to browsable HTML

## 2. Data Schemas and Object Specification

### Session Data Formats

#### JSON Format (Web Sessions via API)
```json
{
  "loglines": [
    {
      "type": "user" | "assistant",
      "timestamp": "ISO8601 string",
      "isCompactSummary": boolean,  // Optional: marks continuation 
summaries
      "message": {
        "role": "user" | "assistant",
        "content": string | ContentBlock[]
      }
    }
  ]
}
```

#### JSONL Format (Local Sessions)
```jsonl
{"type":"summary","summary":"Session description","leafUuid":"uuid"}
{"type":"custom-title","customTitle":"User-defined title","sessionId":"id"}
{"type":"user","timestamp":"ISO8601","sessionId":"id","cwd":"/path","gitBran
ch":"main","message":{...},"uuid":"msg-id"}
{"type":"assistant","timestamp":"ISO8601","sessionId":"id","message":{...},"
uuid":"msg-id"}
```

### Content Block Types

| Block Type | Fields | Purpose |
|------------|--------|---------|
| `text` | `text: string` | Markdown text content |
| `thinking` | `thinking: string` | Claude's reasoning process |
| `tool_use` | `id`, `name`, `input: object` | Tool invocation |
| `tool_result` | `tool_use_id`, `content`, `is_error` | Tool execution 
result |
| `image` | `source: {type, media_type, data}` | Base64-encoded image |

### Tool Input Schemas

**Write Tool:**
```json
{
  "file_path": "/absolute/path/to/file.ext",
  "content": "file contents as string"
}
```

**Edit Tool:**
```json
{
  "file_path": "/absolute/path/to/file.ext",
  "old_string": "text to replace",
  "new_string": "replacement text",
  "replace_all": boolean  // Optional
}
```

**Bash Tool:**
```json
{
  "command": "shell command string",
  "description": "Optional human-readable description"
}
```

**TodoWrite Tool:**
```json
{
  "todos": [
    {
      "content": "Task description",
      "status": "pending" | "in_progress" | "completed",
      "activeForm": "Active verb form"
    }
  ]
}
```

### Internal Data Structures

**Conversation Object:**
```python
{
    "timestamp": str,           # ISO8601 timestamp
    "user_text": str,           # User's prompt text
    "messages": List[Tuple],    # [(log_type, message_json, timestamp), ...]
    "is_continuation": bool     # Whether this is a continuation summary
}
```

**Analysis Stats Object:**
```python
{
    "tool_counts": Dict[str, int],  # {"Bash": 5, "Write": 2, ...}
    "long_texts": List[str],        # Long assistant text blocks
    "commits": List[Tuple]          # [(hash, message, timestamp), ...]
}
```

## 3. Application Flows, Logic, and Control Paths

### Main Execution Flow

```
CLI Entry (main())
    │
    ├── local command (default)
    │   └── find_local_sessions() → Interactive picker → generate_html()
    │
    ├── web command
    │   └── fetch_session_from_api() → generate_html_from_session_data()
    │
    ├── json command
    │   └── parse_session_file() → generate_html()
    │
    └── all command
        └── find_all_sessions() → generate_batch_html()
```

### HTML Generation Pipeline

```
1. PARSE: parse_session_file(path)
   └── Detect format (JSON vs JSONL)
   └── Return normalized {"loglines": [...]} structure

2. EXTRACT: Process loglines
   └── Group into conversations by user prompts
   └── Track continuation summaries
   └── Auto-detect GitHub repo from git push output

3. RENDER: For each conversation
   └── render_message() for each logline
       └── render_content_block() for each content block
           ├── render_markdown_text() for text
           ├── render_write_tool() for Write
           ├── render_edit_tool() for Edit
           ├── render_bash_tool() for Bash
           └── render_todo_write() for TodoWrite

4. PAGINATE: Group conversations (10 per page)
   └── Generate page-XXX.html files

5. INDEX: Build index.html
   └── analyze_conversation() for stats
   └── Extract commits for timeline
   └── Merge prompts + commits chronologically

6. OUTPUT: Write files
   └── Optionally inject gist preview JS
   └── Optionally upload to GitHub Gist
```

## 7. Proposed Tasks and Issues (Updated with UI Evidence)

### Critical

1. Render tool reply content-block arrays as actual rendered content (not 
raw JSON)

* Observed: TOOL REPLY sometimes displays JSON arrays like [{ "type": 
"text", "text": "..." }] directly in the UI.
* Required: Detect when tool_result content is JSON that parses to the 
project’s content-block schema and render it as content (markdown for text, 
code formatting for code, etc.).
* Provide a raw/parsed toggle, but default to parsed.

2. Sanitize terminal outputs (strip or convert ANSI escape codes)

* Observed: terminal output includes visible ANSI sequences (e.g., 
“[38;2;…m”) and other control characters.
* Required: Implement ANSI detection and sanitization so terminal output is 
readable. Baseline: strip ANSI. Optional enhancement: convert ANSI to styled
 HTML.

3. Replace ambiguous “Show more” truncation with a consistent 
collapsible-cell component

* Observed: “Show more” is visually ambiguous and inconsistent across block 
types.
* Required: Standardize collapse/expand behavior using a single reusable 
component with consistent header controls and predictable state.

### Major

4. Structured renderer for Subagent “Task” blocks (instead of raw JSON)

* Observed: Task blocks show JSON with fields like subagent_type and prompt.
* Required: Render these as a structured card:

  * Title: “Subagent Task” + concise subtitle
  * Fields: agent type, prompt (rendered markdown where applicable)
  * Controls: copy prompt, copy raw JSON (optional)

5. Visually pair tool calls with their corresponding tool replies

* Observed: tool interactions appear as separate cards, making execution 
flow hard to follow.
* Required: Group tool_use and tool_result into a single paired unit (call 
above reply) within a collapsible wrapper, with consistent controls and 
anchors.

6. Improve ergonomics for large log blocks

* Observed: huge log content dominates pages.
* Required: default-collapse large blocks by line/character thresholds, add 
wrap toggle, copy, fullscreen, and optional “download as .txt”.

### Minor

7. Strengthen navigation for long transcripts

* Improve pagination usability (sticky top navigation, optional bottom 
navigation).
* Add per-message anchors and “copy link to this message”.
* Optional: “jump to next/previous tool call/result” controls.

8. Accessibility and keyboard navigation polish

* Ensure all interactive controls have ARIA labels, are reachable by 
keyboard, and have visible focus styles.

to inconsistent collapse behavior, lack of tight pairing between tool calls 
and tool replies, and large blocks that overwhelm the scroll.

Key observations:

* Long content blocks (logs, tool outputs, large markdown sections) dominate
 vertical space and reduce scanability.
* “Show more” truncation is present, but it is visually ambiguous and does 
not clearly communicate scope (what expands), state (expanded vs collapsed),
 or offer consistent controls across content types.
* The UI lacks consistent “universal” controls per block (copy, 
expand/fullscreen, raw/parsed toggle, wrap toggle), which makes repetitive 
transcript navigation and inspection slow.

### 2.2 Interaction patterns and truncation behavior (“Show more”)

The current truncation pattern uses a fade/gradient plus a “Show more” 
button at the bottom of certain blocks. Issues observed:

* The “Show more” affordance looks similar across different content types 
(markdown, tool output, code-like blocks), even though the user’s intent 
differs for each.
* It is not obvious whether expanding affects the specific block, the whole 
message, or nested content.
* The state is not clearly persistent: the UI does not visibly convey a 
stable “collapsed/expanded” state per unit, nor does it clearly indicate a 
“Show less” counterpart or persistent memory across reload.

This reinforces the need for a unified collapsible-cell component with 
consistent header controls and predictable behavior.

### 2.3 Data representation issues (highest impact)

Several content types that have well-defined structure are currently 
displayed as raw JSON, which significantly harms usability:

1. Subagent “Task” blocks are rendered as raw JSON

* Example: cards titled “Task” show a JSON object with fields such as 
subagent_type and prompt.
* These should be displayed as a structured, human-readable view with clear 
labeling (agent type, task title/summary, prompt) and copy controls.

2. Tool replies are sometimes rendered as raw JSON arrays of content blocks

* Example: “TOOL REPLY” shows an array like [{ "type": "text", "text": "..."
 }] rather than rendering the text/markdown itself.
* This indicates the rendering pipeline is not consistently interpreting 
tool_result content that is structured as content-block objects.
* The expected UX is to render these blocks as their corresponding display 
types (markdown text, code, etc.), with an optional “raw” toggle for 
debugging.

3. Terminal/ANSI escape codes leak into the UI

* Example: console output includes visible ANSI sequences (e.g., “[38;2;…m”)
 and other control characters.
* This makes output unreadable and must be addressed via stripping or 
converting ANSI to HTML.

### 2.4 Long log and terminal output ergonomics

The screenshots show extremely large pasted terminal logs inside messages. 
Current behavior makes these blocks hard to work with:

* Excessive vertical space with no strong default collapse-by-lines 
behavior.
* No wrap toggle, no “copy” affordance, and no “download as text” 
affordance.
* No quick jump navigation around large outputs.

A dedicated handling strategy is needed for “terminal-like” blocks:

* Detect and sanitize ANSI/control sequences.
* Default-collapse beyond a line/character threshold.
* Provide wrap toggle, copy, and fullscreen expansion for inspection.

### 2.5 Navigation and “where am I?” context

Pagination and an index button exist, but long reading sessions still lack 
key navigation aids:

* There is no visible per-page table of contents (TOC) or per-message 
anchors surfaced as navigation controls.
* There is no “jump to next tool call/result” navigation, which is 
especially valuable in transcripts dominated by tool interactions.
* The interface would benefit from a sticky navigation header (pagination + 
quick-jump + search) and stable anchors per message/tool interaction.

### 2.6 Suggested priorities derived from the screenshots

Based on observed pain points, the highest leverage improvements are:

1. Render structured content (Task blocks, tool replies containing 
content-block arrays) as human-readable UI by default, with a raw toggle.
2. Introduce consistent collapsible-cell behavior and header controls across
 all block types.
3. Sanitize terminal outputs (strip or convert ANSI), and add ergonomics for
 large outputs (collapse thresholds, wrap, copy, fullscreen).
4. Add TOC/anchors and improve navigation for long transcripts.

def clean_escape_chars(text):
    """Remove literal escape sequences and render properly."""
    # Replace literal \n with actual newlines
    text = text.replace('\\n', '\n')
    text = text.replace('\\t', '\t')
    text = text.replace('\\r', '')
    return text


def detect_content_type(content):
    """Detect if content is JSON, code, markdown, or plain text."""
    content = content.strip()
    
    # JSON detection
    if (content.startswith('{') and content.endswith('}')) or \
       (content.startswith('[') and content.endswith(']')):
        try:
            json.loads(content)
            return "json"
        except:
            pass
    
    # Code detection (common patterns)
    code_patterns = [
        r'^(def |class |import |from |function |const |let |var )',
        r'^#!/',
        r'^\s*<[a-z]+[^>]*>',  # HTML/XML
    ]
    for pattern in code_patterns:
        if re.match(pattern, content, re.MULTILINE):
            return "code"
    
    # Markdown detection
    if re.search(r'^#{1,6}\s|\*\*|__|\[.*\]\(.*\)|```', content, 
re.MULTILINE):
        return "markdown"
    
    return "text"
```

def pair_tool_calls_with_results(messages):
    """Match tool_use blocks with their tool_result responses."""
    tool_calls = {}  # id -> tool_use block
    paired = []
    
    for msg_type, msg_json, timestamp in messages:
        msg = json.loads(msg_json)
        content = msg.get("content", [])
        
        if isinstance(content, list):
            for block in content:
                if block.get("type") == "tool_use":
                    tool_id = block.get("id")
                    tool_calls[tool_id] = {
                        "call": block,
                        "timestamp": timestamp,
                        "result": None
                    }
                elif block.get("type") == "tool_result":
                    tool_id = block.get("tool_use_id")
                    if tool_id in tool_calls:
                        tool_calls[tool_id]["result"] = block

// State persistence manager
class UIStateManager {
  constructor(sessionId) {
    this.sessionId = sessionId;
    this.storageKey = `ui-state-${sessionId}`;
  }
  
  save() {
    const state = {
      scrollPosition: window.scrollY,
      collapsedCells: this.getCollapsedCells(),
      activeTab: document.querySelector('.thread-tab.active')?.dataset.view,
      tocVisible: !document.getElementById('table-of-contents').hidden,
      timestamp: Date.now()
    };
    localStorage.setItem(this.storageKey, JSON.stringify(state));
  }
  
  restore() {
    const state = JSON.parse(localStorage.getItem(this.storageKey) || '{}');
    
    if (state.collapsedCells) {
      state.collapsedCells.forEach(id => {
        const cell = document.querySelector(`[cell-id="${id}"]`);
        if (cell) cell.collapsed = true;
      });
    }
    
    if (state.scrollPosition) {
      setTimeout(() => window.scrollTo(0, state.scrollPosition), 100);
    }
  }
  
  getCollapsedCells() {
    return 
Array.from(document.querySelectorAll('collapsible-cell[collapsed]'))
      .map(cell => cell.getAttribute('cell-id'));
  }
}

// Auto-save on visibility change
document.addEventListener('visibilitychange', () => {

    uiState.save();
  }
